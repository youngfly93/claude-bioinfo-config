# 交付目录标准结构（模板 · 由 structure_check.py 机器强制）

客户拿到包要：**结构清晰、一眼定位、无冗余、好找**。交付目录一律按下面摆，`structure_check.py` 会在打包前核对。

```
项目名_交付_YYYYMMDD/            # 解压即此单一根目录（zip_pack 保证，无 delivery 外壳）
├── README.md                   # 唯一说明：分析概览 + 怎么读这个包 + 逐脚本说明 + 联系方式
├── 01_分析报告/                # 正式 Word/PDF 报告（客户第一眼）
├── 02_质控/                    # 一个分析主题 = 一个编号夹，编号即阅读顺序
│   ├── 图/                     # 只放图：png/pdf/tiff/svg
│   └── 表/                     # 只放表：csv/xlsx
├── 03_差异表达/  ├─图/ └─表/
├── 04_富集分析/  ├─图/ └─表/
├── ...                         # 主题按分析逻辑顺序编号递增（02 起）
├── 分析代码/                   # 编号脚本 01_*.R 02_*.R…（命名见名知意；脚本说明在根 README）
└── 溯源表.xlsx                 # 可选：结果/图 → 源数据 → 生成脚本（承重项建议保留）
```

## 硬规则（structure_check 会拦）

- **P1（必拦）放错位置**：`图/` 里出现表（csv/xlsx）、`表/` 里出现图（png/pdf…）、整包没有报告。
- **P2（建议/严格模式拦）**：无 `01_分析报告/`；主题夹没分 `图/表`（图表混放）；根目录有散落文件（放行：根 `README`、`00_*`、溯源表）。
- **P3（提示）**：缺 `溯源表.xlsx`、主题编号跳号。

## 约定（少文档、命名易懂、只留必要）

- **文档只有根 `README.md` 一份**：概览 + 怎么读 + 逐脚本说明全并进去；不产 `00_目录导航.md`、不放子目录 README（多份说明正是"AI 感"来源）。
- 编号从 `02` 起按 质控→差异→富集→… 顺序排，**编号即阅读顺序**。
- 每主题内固定 `图/`、`表/` 两子目录，图表物理隔离；图只进图、表只进表。
- **只收覆盖分析的必要文件**（脚本+图+结果表+报告）；中间文件（.RData/.rds/.log）、原始数据（fastq/bam）、过程记录（proof/audit/.work/HANDOFF/execution_log/草稿）由 `zip_pack.py` 机械排除、不进包。
- 客户表格优先 `.xlsx`（Excel 原生不乱码）；必须 CSV 则写 UTF-8 BOM。
