# 审计-修复一体化循环

对照 `plan.md` 执行穷尽式审计-修复循环。**会改文件**：缺失分析重跑、数据不一致追溯修正、图表重生成、代码直接改。不要只做计划，但只执行用户已授权范围内的修复。

> 只需只读审计、不动文件 → 用 `bio-result-audit`。

## 权限边界

- 用户没有明确授权修改时，先停在 `bio-result-audit` 的只读审计。
- 可以修改分析脚本、结果文件、图表和报告来修复审计问题；每轮修复前先检查 `git status`，不要覆盖无关改动。
- **只在正本目录改**：只动当前项目的正本目录，绝不改 `.tmp*`、旧版本或副本目录；动手前先确认所在路径是正本（项目里常有 `_v123x`、`0613 2/3` 之类副本，极易改错）。
- **删/覆盖数据前先确认**：删除或覆盖任何数据文件前，先展示其内容、说明影响、等确认；承重判断先用第二种独立方法复核，别凭单次读取/正则就删（曾因误判候选数删过同事的对照行）。
- **默认不执行 `git commit`**。只有用户明确说“提交”“commit”“修完提交”时，才提交本轮修复；否则只给出建议提交信息。
- 单个 bug 无法快速定位时，切到 `bio-diagnose` 先建立最小复现反馈回路。

## 审计维度

每轮对每个维度逐一检查：

1. **完整性**：plan.md 要求的每个分析步骤是否都有对应结果文件
2. **数据准确性**：结果文件数据是否与原始/中间文件一致（抽样验证关键数值）
3. **分析准确性**：统计方法、参数、阈值是否正确
4. **方法合理性**：工具/流程/参考基因组版本是否合适
5. **图表-数据一致性**：图表展示的数据是否与结果文件吻合
6. **基因组版本**：参考基因组/注释版本是否与 plan.md 一致

## 问题分级

- **P0 — 数据完整性**：缺失结果、数据丢失、关键文件损坏
- **P1 — 方法正确性**：参数错误、工具版本不对、基因组版本不匹配
- **P2 — 完整性缺失**：部分分析未完成但不影响核心结论
- **P3 — 润色优化**：图表美化、报告格式、命名规范

## 执行流程

### 每一轮：
1. **审计**：用 `bio-result-audit` 全面审计；若用户授权 subagent，可让 subagent 做只读审计并回传问题清单
2. **分级**：按 P0→P3 排序
3. **修复**：立即修复，P0 优先（缺失→重跑脚本；不一致→追溯修正；图表→重生成；代码→直接改）
4. **重审**：验证修复生效、是否引入新问题
5. **可选提交**：仅在用户明确授权提交时执行 `git commit`，提交范围必须只包含本轮修复

### 循环控制
- 最多 **5 轮**
- P0/P1 清零且无新问题 → 提前终止
- 每轮开始输出当前轮次和剩余问题数

### 抗崩溃（防 token 上限/超时丢进度）
- 每轮**实时写盘**：审计清单 → `audit/round_N.md`；本轮改了哪些文件 → `fix_log.md`（追加，含路径 + 动作 + 依据）。
- 用 TaskCreate/TaskUpdate 把每轮、每个 P0/P1 修复登记成检查点，主线程只留简短摘要。
- 委派 subagent 审计时，要求它把完整发现写盘、只回一行摘要（见 `bio-result-audit`）。
- 会话中途死掉 → 从 `fix_log.md` / 最后检查点恢复，只补做未完成项，不从头重跑。

## 最终输出

审计进展矩阵：

```
| 轮次 | P0 | P1 | P2 | P3 | 状态 |
|------|----|----|----|----|------|
| R1   | 3  | 5  | 2  | 4  | 修复中 |
| R2   | 0  | 2  | 2  | 4  | 修复中 |
| R3   | 0  | 0  | 1  | 3  | 通过 ✅ |
```

剩余未修复问题（如有）及原因说明。

若未提交 git，输出建议提交信息和涉及文件清单。
