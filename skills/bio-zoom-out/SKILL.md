# 生信子项目地图

我不熟这块。升一层抽象，给我当前子项目的全局地图——**只读、不改文件、不跑分析**，用项目自己的术语讲，不引入新黑话。

## 先读什么（按这个顺序，能查到的不猜）

1. `plan.md` — 分析步骤与当前进度（最重要，先读）
2. `CONTEXT.md` / `README.md` — 术语、样本命名、项目背景
3. `CLAUDE.md` / `AGENTS.md` — 项目专属指令
4. `scripts/` — 看编号顺序（`01_`、`02_`…）和每个脚本头部注释/文件名，推断管线
5. `data/`、`results/`、`config/` — 看输入（原始 vs 处理）、产出、参数在哪
6. `docs/analysis-decisions/`（若有）— 已记录的"为什么这样分析"

## 输出：一张地图

简洁、可扫读，包含下面几块：

### 1. 一句话定位
这个子项目在回答什么**生物学问题**、属于谁（协作者）、现在处于什么阶段。

### 2. 分析管线（数据流）
按脚本编号顺序，每步一句话："输入 → 做什么 → 输出"。例如：
```
01_data_prep.R     raw counts (GSE12345) → 过滤+QC → se.rds
02_deg.R           se.rds → DESeq2 (treat vs ctrl) → deg.csv
03_enrich.R        deg.csv → GO/KEGG 富集 → 图表
```

### 3. 关键文件位置
plan.md、数据（原始/处理分别在哪）、结果、脚本、配置、关键中间对象（.rds/.RData）。

### 4. 技术栈与运行环境
R / Python / Nextflow / Shiny；关键包；跑在本地还是服务器（<compute-server>）。

### 5. 当前进度
依据 plan.md：哪些步骤 done、哪些 pending、卡在哪。

## 纪律
- **只读**：不执行分析、不修改任何文件。
- **复用已有文档**：优先引用 plan.md/CONTEXT.md 的原话，别重复造一份。
- **诚实标注未知**：plan.md 缺失、脚本无注释导致推断不确定的地方，明确说"推断/不确定"，不要编。
- **指出异常**：发现脚本编号断档、产物与脚本对不上、data 里有可疑重复等，顺手提一句。
