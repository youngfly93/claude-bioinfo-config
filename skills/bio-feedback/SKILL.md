# harness 反馈收档与复盘

harness 还年轻，成熟只能靠真实里程喂——但"用着哪卡了"当场不记，就随会话蒸发。本 skill 把它收档，让以后能读、能聚类、能驱动改进。**它记的是"工具好不好用"，不是"分析对不对"（那是 audit）、不是"做到哪了"（那是 handoff/log）。**

> 设计哲学承 `bio-deliver`：**少即是多·承重才留**。反馈台账的价值全在**真实**——只记真摩擦、别为完整凑数；复盘要改 harness 时，改动本身也过四问门。

## 收档位置（本地、绝不进公开仓）

- 收档夹：`~/.claude/harness-feedback/`（本地运行态；harness 公开仓只有本 skill 协议、**无任何反馈数据**）。
- 一次一文件：`~/.claude/harness-feedback/<YYYY-MM-DD>_<项目本地名>.md`；同项目同日**追加同一份**，别堆多版本。

## 动作一：记档（record）

**时机**：用户说"记反馈"、bio-deliver 交付收尾自动触发、或会话里明显撞到 harness 摩擦时。

1. **agent 起草**——扫**这轮实际发生的**摩擦，逐条**具体**（不是泛泛"体验不错"）：
   - 哪个 skill / checker / hook / 约定卡了？具体怎么错的 / 怎么含糊？我怎么绕过去的？
   - 例："`figure_spec_check` 对 X 误报 P2"、"语气锚找不到作者样本、只能猜"、"打包蓝图在无 `results/` 的项目上不适用"。
   - **只写真摩擦**：这轮顺 → 写"无摩擦"或直接跳过，**绝不编建议凑数、不写自我表扬**。
2. **用户拍板**——删 / 改 / 补**体验级**摩擦（哪 skill 烦、哪步别扭——agent 感觉不到的那部分）。
3. **写一条**到收档夹，字段固定：

```markdown
# harness 反馈 · <项目本地名>  (<日期>)
- harness_commit: <proof.json 的 harness_version / 或 `git -C <harness> rev-parse --short HEAD`>
- 摩擦:
  - [P1/P2/P3] <具体哪里 · 怎么错 · 怎么绕> · 涉及 <skill/checker/hook>
- 建议(可选，别硬凑): <若显而易见的修法>
```

**隐私**：项目用**本地目录名**指代——无客户身份 / 患者数据 / 真实项目学名（收档夹虽本地，也守脱敏习惯，别把公开仓刚清掉的东西又落到别处）。

## 动作二：复盘（review）

**时机**：要优化 harness 前、用户说"看反馈 / harness 哪要改"、攒了若干条时。

1. 读 `~/.claude/harness-feedback/` 全部条目（**纯读 + 判断，无需工具**——一堆短笔记不用脚本）。
2. **聚类**：同一处（skill/checker/hook）被吐槽了几次？按 **频次 × 严重** 排。
3. 产**改进清单**，但每条进清单前**过四问门**：
   - **只有复现摩擦（≥2~3 次）或明显严重的单次**才值得改 harness；一次性抱怨可能是偶然，**先挂着观察、别急着改**（防"一条吐槽加一个补丁"的膨胀）。
   - 能约定解决就别加机械；沉默+灾难才机械化；真源唯一别重述。
4. 输出：`| 摩擦 | 复现次数 | 建议动作 | 是否值得现在改 |`，交用户定。

## 纪律

- **只记真摩擦、不注水、不自夸**；顺就写"无摩擦"。台账一旦被套话填满，整个回路就废了。
- **记工具、不记客户**：无客户数据；项目用本地名。
- **数据只在本地** `~/.claude/harness-feedback/`，绝不进公开仓；本 skill 公开的只是**协议**。
- **复盘 ≠ 照单全改**：一次性摩擦先观察，复现的才动手，且改动过四问门（否则反馈回路自己会把 harness 喂胖）。
- 与 `bio-handoff`(状态) / `execution_log`(流水) / `bio-result-audit`(结果对错) 分工不混——本 skill 只管**"harness 自身好不好用"**这一件事。
