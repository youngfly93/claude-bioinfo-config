# 生信结果审计

对照 `plan.md` 审计现有分析结果。默认**只读**：不重跑完整分析、不改文件、不删除数据。目标是找出完成度、数据准确性、分析准确性、方法合理性和交付风险。

## 执行模式

### Subagent 优先（当前环境允许且用户授权时）

如果用户明确要求使用 subagent、子代理、委派审计，或当前会话已被用户设置为“bio-result-audit 默认用 subagent”，优先把审计任务交给一个只读 subagent。主线程不重复审计细节，只负责：

- 提供清晰任务边界：项目路径、`plan.md` 路径、只读约束、输出格式。
- 在 subagent 审计时处理非重叠工作（例如准备交付清单或等待用户补充上下文）。
- 审阅 subagent 结果，整合成最终报告；必要时追问用户或要求补充证据。

推荐 subagent 任务提示：

```text
按 bio-result-audit 的规则，对当前生信项目做只读审计：
1. 读取 plan.md，提取计划任务、数据类型、分组/contrast、阈值、交付物和参考版本。
2. 扫描 results/ figures/ reports/ notebooks/ output/ analysis/ 和编号目录。
3. 追溯关键结果到脚本、notebook、pipeline、日志或配置。
4. 检查完整性、数据准确性、分析准确性、方法合理性、图表-数据一致性和参考基因组/注释版本。
5. 对承重的数字和结论做针对性复算与数字台账（见工作流第 6 步）。
5b. 方法保真三反射（见工作流 6B）：每个 not_run/not_assessable/missing_X 逐个核实 blocker 真伪（理由不实=P1）；受控词表列查 raw→mapped 折进率；fallback 分「试过失败/从没试/授权延期」。产出方法保真表。
6. 输出结构化审计报告，按 P0/P1/P2/P3 排序问题。
7. 把每步完整发现实时写进 `audit/<module>.claude.md`（module 用 plan.md 任务名逐字、头记 audited_commit），只向主线程回一行状态摘要（防撞输出上限/超时丢进度，便于只补跑缺失模块）。

严格只读：不重跑完整分析，不修改、删除、移动文件，不提交 git。
所有结论必须标明事实依据、推断或建议。
```

### 本地执行 fallback

如果当前工具环境没有 subagent，或用户没有授权子代理，主线程直接按下面工作流执行审计。

## 工作流

### 1. 读取计划

- 优先读当前项目的 `plan.md`。
- 若没有 `plan.md`，查找类似计划文档；仍找不到就停止并询问用户。
- 提取：分析目标、数据类型、物种/参考版本、分组/contrast、预期结果、交付物、阈值和关键 QC 断言。

### 2. 扫描结果

只做低风险只读扫描：

- 结果目录：`results/`、`figures/`、`reports/`、`notebooks/`、`output/`、`analysis/`、编号目录如 `01_*`、`02_*`。
- 文件类型：`.csv`、`.tsv`、`.xlsx`、`.png`、`.pdf`、`.svg`、`.html`、`.docx`、`.ipynb`、日志和 pipeline 输出。
- 记录每类结果的位置、文件数、最近修改时间、基本尺寸/行列数（可快速读取时）。

### 2.5 步骤地图 + 清爽体检（只用 Glob/Grep，无需跑脚本）

让审查一眼看清「各步是什么、脚本+数据是什么、路径是否合理」，并揪出多余中间文件。项目按 `bio-project-init` 约定时（脚本头四行契约 + `results/<NN_step>/` 编号对齐）尤其高效：

- **重建步骤地图**：`Grep '^# (步骤|上游|输出|种子)'` 扫 `scripts/analysis/`、`scripts/figures/`，拼出每步「脚本 → 上游输入 → 输出」链。缺契约头的脚本 = 不可溯源，记 P2。
- **验路径合理性**：每个 `# 输出:` 声明的文件是否真存在、是否落在编号对齐的 `results/<NN_step>/`；每个 `# 上游:` 是否存在且指向上游某步产物（而非原始数据被绕过）。声明与落盘不符 = 发现项。
- **揪孤儿/多余中间文件**：`Glob results/**`、`figures/**`，凡**没有任何脚本 `# 输出:` 声明它**的文件 = 孤儿；可再生的中间物/诊断图若躺在 `results/` 而非 `.work/` = 多余。二者都作为「清爽」类发现列出（只标记，不顺手删）。
- **绘图零重算校验**：`Grep` 绘图脚本里是否直接读原始数据或现算统计量（`read.*counts`、`DESeq\|wilcox\|t.test` 等出现在 `scripts/figures/`）= 违反「图只消费 results/」，记发现。

### 3. 结果溯源

对关键结果追溯来源：

- 查找相关 `.R`、`.py`、`.sh`、`.Rmd`、`.ipynb`、`Snakefile`、`.nf`、配置文件和日志。
- 读取脚本核心逻辑，确认输入、输出、参数、阈值、参考版本、软件版本。
- 明确区分**事实**、**推断**和**建议**。

### 4. 覆盖度检查

逐项对照 `plan.md`：

- `✅ 已完成`：有对应完整结果，来源清楚。
- `⚠️ 部分覆盖`：做了一部分、命名/格式偏差、关键产物缺失。
- `❌ 未见结果`：没有找到对应输出或无法溯源。

### 5. 合理性与风险

重点检查：

- 数据准确性：样本数、分组、行列数、关键数值是否与源数据/中间文件一致。
- 分析准确性：contrast 方向、阈值、统计模型、协变量、批次处理是否符合计划。
- 方法合理性：工具与数据类型是否匹配；是否把技术重复当生物重复；count/TPM/log 值是否误用。
- 统计合规：多重检验是否做了校正（FDR/BH）；样本量/检验力是否够；检验方法是否匹配数据分布；批次/混杂是否纳入模型；cutoff 是否数据驱动过拟合。**按分析类型可跑 harness 确定性 validator 早筛**：`${CLAUDE_PLUGIN_ROOT:-$HOME/.claude}/harness/quality/{bulk_rnaseq,clinical_survival}_validator.py .`（查 contrast 方向/分组重复/FDR-logFC 阈值/gene id；删失编码/时间单位/cutpoint/PH 假设）。
- 版本合理性：参考基因组、注释版本、数据库版本是否与 `plan.md` 或项目背景一致。
- 图表一致性：图中数值、样本数、分组、显著性是否能追溯到结果表。**客观项（分辨率/白底/DPI）跑 `fig_check.py`、docx 跑 `docx_check.py` 别肉眼估**；图表专项审查转 `bio-fig-review`（含 house 样式一致性）。

### 6. 关键 claim 复算与数字台账（交付/临床/承重结论必做）

承重的数字和结论不能只“溯源到表”，要**针对性地从源数据把这一个量重算一遍**（针对单个 claim，不是重跑整条分析流程）再逐条比对，重点抓最容易翻车的几类：

- **方向/符号**：effect、logFC、HR/OR、相关性的正负——结论写反是最贵的错。
- **统计量**：p 值、FDR、NRI/IDI、bootstrap/置换结果，量级能否复现（曾出现 bootstrap p 值 bug）。
- **逐字对账**：报告里每个数与源表单元格字字对得上，样本数/分组一致。

产出一张**数字溯源台账**，承重 claim 逐条列：

| 报告中的 claim | 报告位置 | 源数据（文件:列/单元格） | 复算值 | 一致? |
|---|---|---|---|---|

任一不一致 → 记 P0/P1，在审计报告里单独点名。

### 6B. 方法保真三反射（复算之外必做——复算只证"数字非造假"，证不了"方法对"）

⚠️ 被偷换/跳过的方法会产出**真实、可复现**的数字，复算照样逐位对上。故每模块在数字台账之外再过三关（数值复算的盲区、漏判高发区）：

1. **理由核实（reason-truthing）**：每个 `not_run/not_assessable/not_testable/no_X_available/missing_X/blocked/skipped/fallback/deferred` 都是**待验证 claim 非事实**，逐个到独立源头验 blocker 真伪（例：称"缺 subject_id"→ grep 源表确认该数据集真缺该列；称"包缺失"→ 确认是否独立包而非找错命名空间）。理由对某场景根本不成立 = P1，且改正错误标注。
2. **raw-保真 > 自洽**：门控/映射回溯 **RAW 源字段**，拒同源派生量自证（`record_count==sample_count` 是同义反复非 metadata_match_rate）；受控词表列（site/age/disease）查 **raw→mapped 折进率**抓静默坍缩。
3. **fallback 三分类**：缺失/降级分「**试过失败**(诚实边界)/**从没试**(静默降级)/**授权延期**」——只有"试过撞墙"证据才算诚实边界；"从没试"包装成边界或图件缺失 = 降级(P2 起)。

> **fitness-for-purpose（另一条轴，独立于上面的"诚实度"）**：上面三关判"诚实/方法对不对"，但**诚实边界照样能产出下游不可用的交付物**。所以每个降级再问一层：**它是否破坏下游可用性？** **判据锚 `plan/spec`，不是审计员主观**：`fit` = 满足 `plan/spec` **为它声明的那个下游用途**；plan 没要求的标准**不能**拿来判 unfit（否则 fitness 变成新假阳面）；fitness 判定同样要**给证据(file:line) + 指向 plan 声明的用途**（否则它就是这套"证据优先"体系里的主观孤岛）。（例：scRNA 因缺包未做 doublet/未聚类 → 注释与 signature 取自 pre-QC raw、不可信、进不了 plan 声明的 stage2 单细胞准入；批次**与生物学混杂且未建模** → 差异不可信。注：批次不混杂 / 已作协变量建模 / plan 本就没要求去批次时**不校正反而对**，别误判成破坏）。**破坏下游 fitness 的，无关是否如实披露，升 P1**——别因"诚实边界"就当 P3 放行。**阻塞范围看它是否 `plan/spec` 声明的本阶段/下游准入必需件**：必需件有缺陷 → 阻塞该阶段放行；不在关键路径上的不合格件 → 不必阻塞全交付，但**须显式标注不可用、不得当合格件交付**。`present`（在不在）≠ `fit`（够不够进 plan 声明的下游用途），交付物核对两个都要答。

机械兜底别只肉眼：`${CLAUDE_PLUGIN_ROOT:-$HOME/.claude}/harness/quality/limitation_register.py .`（逐个 limitation 要独立证据）、`.../mapping_fidelity.py`（受控词表 raw→mapped 折进率）。产出「方法保真表」：mandated 方法 → 实际 status → [严格/诚实边界/授权override/未披露降级/理由不实] + 证据（spec 锚定逐条、不抽样）。

## 输出格式

```markdown
# 生物信息学分析审计报告

## 1. 总体评价
[现有结果与 plan.md 的符合度；不要空泛表扬]

## 2. Plan.md 需求覆盖情况
| 序号 | 计划任务 | 覆盖状态 | 对应结果/证据 | 风险 |
|------|----------|----------|----------------|------|

## 3. 结果溯源与方法评估
### 3.1 [结果类别]
- 来源：[脚本/日志/文件路径]
- 核心逻辑：[输入 -> 方法 -> 输出]
- 评估：[合理/有风险/不合理，说明依据]

## 4. 主要问题与风险
按 P0-P3 排序：
- P0 数据完整性：缺失结果、数据丢失、关键文件损坏
- P1 方法正确性：参数错误、工具/版本不对、contrast/基因组版本不匹配
- P2 完整性缺失：部分分析未完成但不影响核心结论
- P3 润色优化：图表、报告格式、命名规范

## 4B. 方法保真表 + 限制登记（防"数字对但方法被偷换/理由不实"）
| 模块/子步 | mandated 方法 | 实际 method_status | 判定 | 证据 |
|---|---|---|---|---|
判定 ∈ 严格完成/诚实边界/授权override/未披露降级/理由不实；后两类必进 §4(P1)。
限制登记：逐条列 not_run/not_assessable/missing_X + blocker 是否经独立源头核实。

## 5. 建议动作
[具体到文件、脚本或分析步骤；不自动修改]
```

## 纪律

- 有依据地指出问题，不粉饰。
- 所有结论都要能指向路径、脚本、表格、日志或明确推断。
- 只运行只读/极低风险命令；不要重跑完整分析（针对单个承重 claim 的复算除外）。
- **二次复核**：承重结论先用第二种独立方法确认再下判断——单次读取/正则容易误报，曾因误判候选数删过对照行。
- **理由核实**：每个"做不了/not_assessable/缺 X/包缺失"都是待验证 claim——到独立源头验 blocker 真伪，理由不实=P1（把"不采信自报数字"扩到"不采信自报借口"）。
- **raw-保真优于自洽**：门控/映射回溯 RAW 源字段、拒同源派生量自证；受控词表跑 `harness/quality/mapping_fidelity.py`、limitation 跑 `limitation_register.py`，别只肉眼。复算只证不造假、证不了方法对。
- **改/删数据前先看清再确认**：审计只读；若进入修复，只在正本目录改，删除或覆盖任何数据文件前先展示其内容、说明影响、等确认。
- 如果用户要求“审计并修复”，先完成审计，再让用户确认修复范围；不要自动提交 git。
