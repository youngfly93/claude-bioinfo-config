# 生信结果审计

对照 `plan.md` 审计现有分析结果。默认**只读**：不重跑完整分析、不改文件、不删除数据。目标是找出完成度、数据准确性、分析准确性、方法合理性和交付风险。

## 执行模式

### Subagent 优先（当前环境允许且用户授权时）

如果用户明确要求使用 subagent、子代理、委派审计，或当前会话已被用户设置为“bio-result-audit 默认用 subagent”，优先把审计任务交给一个只读 subagent。主线程不重复审计细节，只负责：

- 提供清晰任务边界：项目路径、`plan.md` 路径、只读约束、输出格式。
- 在 subagent 审计时处理非重叠工作（例如准备交付清单或等待用户补充上下文）。
- 审阅 subagent 结果，整合成最终报告；必要时追问用户或要求补充证据。

推荐 subagent 任务提示：

```text
按 bio-result-audit 的规则，对当前生信项目做只读审计：
1. 读取 plan.md，提取计划任务、数据类型、分组/contrast、阈值、交付物和参考版本。
2. 扫描 results/ figures/ reports/ notebooks/ output/ analysis/ 和编号目录。
3. 追溯关键结果到脚本、notebook、pipeline、日志或配置。
4. 检查完整性、数据准确性、分析准确性、方法合理性、图表-数据一致性和参考基因组/注释版本。
5. 对承重的数字和结论做针对性复算与数字台账（见工作流第 6 步）。
6. 输出结构化审计报告，按 P0/P1/P2/P3 排序问题。
7. 把每步完整发现实时写进 `audit/<模块名>.md`，只向主线程回一行状态摘要（防撞输出上限/超时丢进度，便于只补跑缺失模块）。

严格只读：不重跑完整分析，不修改、删除、移动文件，不提交 git。
所有结论必须标明事实依据、推断或建议。
```

### 本地执行 fallback

如果当前工具环境没有 subagent，或用户没有授权子代理，主线程直接按下面工作流执行审计。

## 工作流

### 1. 读取计划

- 优先读当前项目的 `plan.md`。
- 若没有 `plan.md`，查找类似计划文档；仍找不到就停止并询问用户。
- 提取：分析目标、数据类型、物种/参考版本、分组/contrast、预期结果、交付物、阈值和关键 QC 断言。

### 2. 扫描结果

只做低风险只读扫描：

- 结果目录：`results/`、`figures/`、`reports/`、`notebooks/`、`output/`、`analysis/`、编号目录如 `01_*`、`02_*`。
- 文件类型：`.csv`、`.tsv`、`.xlsx`、`.png`、`.pdf`、`.svg`、`.html`、`.docx`、`.ipynb`、日志和 pipeline 输出。
- 记录每类结果的位置、文件数、最近修改时间、基本尺寸/行列数（可快速读取时）。

### 3. 结果溯源

对关键结果追溯来源：

- 查找相关 `.R`、`.py`、`.sh`、`.Rmd`、`.ipynb`、`Snakefile`、`.nf`、配置文件和日志。
- 读取脚本核心逻辑，确认输入、输出、参数、阈值、参考版本、软件版本。
- 明确区分**事实**、**推断**和**建议**。

### 4. 覆盖度检查

逐项对照 `plan.md`：

- `✅ 已完成`：有对应完整结果，来源清楚。
- `⚠️ 部分覆盖`：做了一部分、命名/格式偏差、关键产物缺失。
- `❌ 未见结果`：没有找到对应输出或无法溯源。

### 5. 合理性与风险

重点检查：

- 数据准确性：样本数、分组、行列数、关键数值是否与源数据/中间文件一致。
- 分析准确性：contrast 方向、阈值、统计模型、协变量、批次处理是否符合计划。
- 方法合理性：工具与数据类型是否匹配；是否把技术重复当生物重复；count/TPM/log 值是否误用。
- 统计合规：多重检验是否做了校正（FDR/BH）；样本量/检验力是否够；检验方法是否匹配数据分布；批次/混杂是否纳入模型；cutoff 是否数据驱动过拟合。**按分析类型可跑 harness 确定性 validator 早筛**：`${CLAUDE_PLUGIN_ROOT:-$HOME/.claude}/harness/quality/{bulk_rnaseq,clinical_survival}_validator.py .`（查 contrast 方向/分组重复/FDR-logFC 阈值/gene id；删失编码/时间单位/cutpoint/PH 假设）。
- 版本合理性：参考基因组、注释版本、数据库版本是否与 `plan.md` 或项目背景一致。
- 图表一致性：图中数值、样本数、分组、显著性是否能追溯到结果表。**客观项（分辨率/白底/DPI）跑 `fig_check.py`、docx 跑 `docx_check.py` 别肉眼估**；图表专项审查转 `bio-fig-review`（含 house 样式一致性）。

### 6. 关键 claim 复算与数字台账（交付/临床/承重结论必做）

承重的数字和结论不能只“溯源到表”，要**针对性地从源数据把这一个量重算一遍**（针对单个 claim，不是重跑整条分析流程）再逐条比对，重点抓最容易翻车的几类：

- **方向/符号**：effect、logFC、HR/OR、相关性的正负——结论写反是最贵的错。
- **统计量**：p 值、FDR、NRI/IDI、bootstrap/置换结果，量级能否复现（曾出现 bootstrap p 值 bug）。
- **逐字对账**：报告里每个数与源表单元格字字对得上，样本数/分组一致。

产出一张**数字溯源台账**，承重 claim 逐条列：

| 报告中的 claim | 报告位置 | 源数据（文件:列/单元格） | 复算值 | 一致? |
|---|---|---|---|---|

任一不一致 → 记 P0/P1，在审计报告里单独点名。

## 输出格式

```markdown
# 生物信息学分析审计报告

## 1. 总体评价
[现有结果与 plan.md 的符合度；不要空泛表扬]

## 2. Plan.md 需求覆盖情况
| 序号 | 计划任务 | 覆盖状态 | 对应结果/证据 | 风险 |
|------|----------|----------|----------------|------|

## 3. 结果溯源与方法评估
### 3.1 [结果类别]
- 来源：[脚本/日志/文件路径]
- 核心逻辑：[输入 -> 方法 -> 输出]
- 评估：[合理/有风险/不合理，说明依据]

## 4. 主要问题与风险
按 P0-P3 排序：
- P0 数据完整性：缺失结果、数据丢失、关键文件损坏
- P1 方法正确性：参数错误、工具/版本不对、contrast/基因组版本不匹配
- P2 完整性缺失：部分分析未完成但不影响核心结论
- P3 润色优化：图表、报告格式、命名规范

## 5. 建议动作
[具体到文件、脚本或分析步骤；不自动修改]
```

## 纪律

- 有依据地指出问题，不粉饰。
- 所有结论都要能指向路径、脚本、表格、日志或明确推断。
- 只运行只读/极低风险命令；不要重跑完整分析（针对单个承重 claim 的复算除外）。
- **二次复核**：承重结论先用第二种独立方法确认再下判断——单次读取/正则容易误报，曾因误判候选数删过对照行。
- **改/删数据前先看清再确认**：审计只读；若进入修复，只在正本目录改，删除或覆盖任何数据文件前先展示其内容、说明影响、等确认。
- 如果用户要求“审计并修复”，先完成审计，再让用户确认修复范围；不要自动提交 git。
